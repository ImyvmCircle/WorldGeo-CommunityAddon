# 20260221Settings系统升级

第一位是要遵循`agents/instructions/development.instructions.md`，然后，请根据任务上下文及任务描述，完成任务。

## 任务上下文

1. `Settings`根本上是本项目的核心依赖`ImyvmWorldGeo`中`Region`和其参数所带有的`GeoScope`中的一个参数。它有三个维度：

   - 一是存在于`Region`和`GeoScope`两个区域的层次上，
   - 二是有针对全层次和针对某个玩家之区别，
   - 三是具有三种分类，权限，效果和游戏规则。其中，权限是指玩家主动操作的能力，效果是玩家或实体会被动获得的游戏中效果，游戏规则涉及本层次划定之范围内世界的属性

   维度冲突时或者Setting项未设置时范围会以一个读取优先级或一个默认的Setting状态来解决。

2. 目前只有三个权限种类的`Settings`得到了实现。所以我们的任务只以这目前的三个任务为实现对象即可，但是要求具有充分的扩展性，不仅能添加新的`Permission`，还要能添加别的任何形式的`Setting`。

## 任务内容

### 前置工作

将与飞行权限相关的任何内容从本项目的一切接入点、表单、机制和语言文件等处删除。暂时不涉及飞行权限的任何考量。

### 自动授权

直觉上讲，所谓默认权限值，就是原则上聚落在世界某一个范围内的存在未对其他玩家在这个范围内的操作产生影响。如果现有的权限值被添加且与这个默认权限值不同，则意味着这项更改会影响，主要是限制，玩家在世界内这个范围的该项操作。吸纳了新成员，那么这个成员应该要不被这项限制所影响。所以，需要添加一个玩家加入聚落时候的自动授权系统。

但是，需要考虑的另一个方面是，有些子区域权限设置就是管理员希望限制聚落内玩家的“特许状”区域，因此修改的作用面也不可能过大。所以，综合以上两项，**自动授权**工作的触发条件归纳为：

- 一个新玩家加入聚落；
- 聚落拥有（一个或几个）`Region`层次的全局的权限设定，且该项设定的内容与（完全没设定时）默认获取到的该权限值不同；

则，对于上述权限设定中，所有没有针对这个新玩家的`Region`层次的特殊设定项的所有项目，增加一个`Region`层次的、针对该玩家的设定项，项目内容为默认值。

例如，默认建筑破坏权限是授予所有玩家了的。如果全局、Region层次上有一个建筑破坏权限为false，那么给新加入玩家一个`Region`层次上的true权限。

必须谨记：

- 不是所有权限默认都是`true`。
- 查看权限到底是什么是有办法的，去`ImyvmWorldGeo`中可以查看每个权限的构造方法细节，然后你就会明白。
- 不要赋予`GeoScope`层次的任何权限，自动授予一定要针对单个玩家
- 如果已经有对应的针对该玩家的，`Region`层次的设置，不要自动授权。

### 定价系统

定价系统的原则是，对改变了原有游戏逻辑的默认状态的行为定价，对波及范围和影响程度定价。定价的时候的波及范围只假设添加了在这个层次上添加这个设定的情况，不考虑其他层次和玩家的特定设定可能对象范围重叠并进一步有可能因优先级而对其效力消除的可能。

#### 权限定价

该类定价的前提条件是：

- 是权限。
- 针对的是一个范围，且其范围不论是`Region`层次还是`GeoScope`范围，面积大于0. 谨记，如果是`Region`层次，不管下一层次有没有对应项已经设置（导致其在一个多个`GeoScope`实际上不能生效)，都简单只计算整个`Region`面积，反之亦然。
- 执行此次设定之后，无论是添加还是修改，都造成其权限值与默认值出现差异。

定价公式：
$$
基础收费 + 权限种类系数 \times 涉及面积大小 \times 操作对象系数
$$
其中：

- 基础收费见下表

| 范围层次\聚落种类 | Manor | Realm |
| ----------------- | ----- | ----- |
| Region            | 200   | 100   |
| Scope             | 100   | 50    |

- 权限种类系数请做成一个放在Config同一个文件夹里的新的配置文件，容以后慢慢配置。这个配置文件以后除种类系数之外，还要放置别的系数。暂时设定为BUILD_BREAK 每一万格200， CONTAINER 每一万格 400。请用合适的系数表示方法减少浮点数精度问题。请把这个新的Config写进DEVELOPEMENT_GUIDELINES.md的对应条目
- 涉及面积大小见定价的最基本说明。
- 操作对象系数即全局1，单个玩家0.2.

### 操作与显示

在菜单里执行一个操作时，参考修改传送点等，同样需要将操作对象、价格及计算依据等制作成一个完整的“订单”发送给执行操作的玩家，并创立一个`PendingOperation`，等待玩家在这个可交互消息中选择确认或取消。可交互消息靠命令实现。

确认执行操作之时，先通过`ImyvmWorldGeo`的api检查是否已有完全层次、对象、内容完全相同的`Setting`项目存在，再检查聚落资产是否足够。如果检查通过，则扣钱，并调用`ImyvmWorldGeo`的Api执行操作

执行操作成功时，向聚落所有正式成员发送消息。
